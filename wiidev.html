Nintendo Wii Development

The Nintendo Wii was part of a trend in the late 2000's where every home console would use PowerPC for processing power. The Playstation 3 and the Xbox 360 were part of this trend as well, since PowerPC architecture was more affordable to use rather than brands listed for maximum performance which obviously cost more. The controllers, dubbed Wiimotes, have a BroadCom BCM2042, which helps it become a Bluetooth esque device complete with the accelerometer and infrared camera to point at the sensor bar. The console's main CPU dubbed Broadway, runs at 729 mhz. It is mostly carried from the CPU of the Nintendo Gamecube, as they both use PowerPC instead this processor was rebranded by IBM at the time.

This diagram shows that there is another CPU for the input/output system. This CPU is dubbed Starlet and it is connected to the various hardware devices in the console, running at 243 mhz in the GPU card (Hollywood). The sole reason why this is a different CPU is because it has its own RAM, seperating it from the main Broadway chip. Starlet is usually responsible for migrating access between many of the firmware devices between it and Broadway, which also grants and denies access to the NAND where the OS data and user files are stored. Since Starlet can be reprogrammed with the same input/output system, it keeps backwards compatibility with the Wii. The Advanced Microcontroller Bus Architecture used wires up several buses like the NAND interface and USB 2.0 controllers. Note that the AES and SHA-1 modules are used for security regarding the NAND and anti piracy. That being said, Starlet has an operating system, one on Broadway and another on Starlet. Both are on the NAND memory. From the start of turning on the power, Starlet executes the OS called the Input/Output Operating System (IOS), which includes a Microkernel to control the ARM9 CPU and communicate with other hardware, drivers to communicate with said hardware and a cryptographic core to accelerate encryption operations. Through IPC, Broadway and Starlet communicate with each other with reserved registered. When IOS is updated, the newer version is added into a 'slot' in which all IOS versions are divided in the NAND. The Starlet chip boots a MIOS when a Gamecube disc is detected, it's almost like a Gamecube emulator inside the Wii.

Broadway runs the infamous System Menu, and from here, the user could run applications, check the message board, run Gamecube and Wii discs. Each Wii has a unique SEEPROM value burned in by Nintendo which allows it to send and recieve messages. It is also possible to change system settings through this OS. When an application is loaded, Starlet calls the respective IOS version to run said program. The Wii has a filesystem in the NAND in which directories are responsible for holding certain data. The /dev folder has all the system components, the aes encryption methods, kernel, etc. /title is used for holding titles installed on the system, all of which have their own ID number. I suggest checking out the Filesystem page on WiiBrew if you want to view this further.
